\section{Methodology}
\label{sec:methodology}

This section describes the three system variants: the RAG system
(Section~\ref{sec:rag-system}), the prompt-only baseline
(Section~\ref{sec:prompt-only}), and the hybrid engine
(Section~\ref{sec:hybrid}). We begin with an overview of the shared
architecture.

% ─────────────────────────────────────────────────────────────────────────────
\subsection{System Overview}
\label{subsec:system-overview}

We developed three system variants that share the same generative model
and output format but differ in how they construct the context given to
the model. Two of the three (RAG and Hybrid) follow the
retrieval-augmented generation paradigm~\cite{lewis2020rag}, while the
third (Prompt-Only) relies entirely on prompt engineering. The shared
infrastructure is organized as a five-stage pipeline tailored to the
Tunisian Bac domain; the retrieval and generation stages are where the
three variants diverge.
Figure~\ref{fig:system-architecture} shows the complete pipeline.

\subsubsection{Pipeline Stages}

\paragraph{Stage~1: Data Acquisition (Section~\ref{subsec:data-acquisition}).}
The corpus is assembled from three categories of material: corrected past
examinations, chapter-based exercise series with worked corrections, and
course notes with theorem statements. Each exercise is acquired
\textit{together with its correction} as a single logical unit. The final
corpus comprises 624 source files spanning 20 chapters.

\paragraph{Stage~2: Digitization (Section~\ref{subsec:digitization}).}
Scanned images and PDFs are converted into \LaTeX{} using Gemini as a
multimodal OCR engine. The pipeline is incremental and resume-friendly:
previously processed files are skipped.

\paragraph{Stage~3: Indexing (Section~\ref{subsec:database-construction}).}
The \LaTeX{} files are normalized, split using adaptive chunking (1{,}500
characters for corrections, 3{,}000 for course material), and embedded with
BGE-M3~\cite{chen2024bge_m3}. Each chunk is annotated with metadata
(document type, chapter, year, session, exercise ID, solution flag)
extracted from the file path. The resulting 1{,}422 chunks are stored in
ChromaDB~\cite{chromadb2024}. An incremental manifest ensures only modified
files are re-embedded.

\paragraph{Stage~4: Two-Stage Retrieval (Section~\ref{subsec:retrieval-engine}).}
The first pass targets corrections (\texttt{is\_solution=true}), seeking
style exemplars. If no close match is found (L2 distance $> 1.2$), a
second pass retrieves course material for theorem backing. A companion fetch
mechanism then locates the exercise statement for each retrieved correction,
ensuring complete exercise--correction pairs in the context.

\paragraph{Stage~5: Generation.}
Retrieved documents are compiled into a prompt with the student's question
and the ``Monsieur Tounsi'' persona. Generation uses Gemini via Vertex AI
($T = 0.15$, max 4{,}096 tokens). Each answer includes a confidence level
and full source transparency.

\subsubsection{Shared Infrastructure}

All three system variants share:
\begin{itemize}
    \item The same generative model (Gemini, $T = 0.15$, 4{,}096 tokens)
    \item The same persona, syllabus guard, and output format
    \item The same Streamlit-based bilingual interface
\end{itemize}

The \textit{only} difference is how each constructs the context: the RAG
system includes retrieved documents, the prompt-only baseline encodes the
curriculum directly in the prompt, and the hybrid engine selects between
the two based on retrieval confidence. This controlled design ensures
that performance differences can be attributed to the retrieval mechanism.

\subsection{Prompt-Only Baseline}
\label{sec:prompt-only}

\subsubsection{Role and Rationale}

The prompt-only baseline is a zero-retrieval system that serves as a control
condition. It receives the same question and generates an answer using the
same Gemini model, temperature, and output budget as the RAG system, but
never queries the vector store. Its entire knowledge of the Bac curriculum
is encoded in the system prompt.

The rationale is straightforward. LLMs already possess substantial
parametric knowledge of standard math
topics~\cite{brown2020gpt3, team2023gemini}, and structured prompts can
elicit reasoning chains that rival more complex
pipelines~\cite{wei2022chain_of_thought}. Without this baseline, we cannot
distinguish what retrieval adds beyond what the model already knows.

The baseline shares as much infrastructure as possible with the RAG system:
same persona, same syllabus guard, same output format, same API
(\texttt{query(question, mode)}). The only difference is the knowledge
source.

\subsubsection{Prompt Architecture}

The system prompt has ten functional components, following the principle of
prompt patterns~\cite{white2023prompt_patterns}:

\begin{description}
    \item[Component~1: Persona.] ``Monsieur Tounsi,'' a fictional veteran
    Tunisian mathematics teacher with 20+ years of Bac preparation
    experience. The persona is defined with specific traits: he speaks
    formal French for mathematical content but uses Tunisian Derja for
    encouragement and motivation (what we call the ``Derja sandwich''
    style), he follows the official correction methodology strictly, and
    he never uses methods outside the Bac syllabus. Role-based prompts
    improve task adherence~\cite{white2023prompt_patterns}.

    \item[Component~2: Curriculum inventory.] All ten chapters with their
    specific topics. This is the core knowledge component---without retrieval,
    the prompt must constrain \textit{which} content the model draws on.
    Compiled from the official program and verified against our corpus.

    \item[Component~3: Reasoning guardrails.] A six-step mental checklist
    (identify, recall, verify, plan, solve, check), adapted from
    chain-of-thought prompting~\cite{wei2022chain_of_thought}.

    \item[Component~4: Output structure.] The six-part answer format
    (identification, rappel du cours, m\'{e}thode, application, conclusion,
    v\'{e}rification) and \LaTeX{} conventions.

    \item[Component~5: Hallucination control.] More aggressive than in the
    RAG system since there is no retrieved context to anchor answers. The
    model must never cite specific exercise numbers or page references
    without certainty.

    \item[Component~6: Syllabus guard.] Forbidden methods (L'H\^opital,
    Taylor series, diagonalization, etc.) with instructions to redirect to
    the Bac-compatible alternative.

    \item[Component~7: Derja sandwich.] Each answer opens and closes with
    a motivational phrase in Tunisian Arabic, with the math body in formal
    French---reflecting how Tunisian teachers actually interact with
    students.

    \item[Component~8: Self-verification.] A visible verification section
    at the end of each answer, forcing the model to check its own work.

    \item[Component~9: Refusal policy.] Explicit refusal for out-of-scope
    questions.

    \item[Component~10: Confidence calibration.] Three confidence labels
    (high, medium, low) based on question characteristics, since no
    retrieval distance is available.
\end{description}

This architecture emerged iteratively during development as we identified
failure modes (curriculum violations, hallucinated references, inconsistent
formatting) and added targeted components to address each one.


\subsubsection{Confidence Without Retrieval}

Without retrieval distances, the prompt-only system uses a keyword-based
heuristic: questions containing standard Bac terms (``r\'{e}currence,''
``limite,'' ``int\'{e}grale,'' ``PGCD'') get high confidence; short or
ambiguous questions get low; the rest default to medium. This is a pragmatic
approximation, not a principled calibration method.

\subsection{Hybrid Engine}
\label{sec:hybrid}

\subsubsection{Motivation}

The RAG and prompt-only systems represent two extremes: always retrieve
(even when matches are poor) vs.\ never retrieve (even when good matches
exist). Recent work on adaptive retrieval---CRAG~\cite{yan2024practical_rag}
and Self-RAG~\cite{asai2023self_rag}---has shown that deciding
\textit{whether} to retrieve can be as important as the retrieval itself.

The hybrid engine implements a simpler version of this idea: it routes each
query to one of three strategies based on the L2 distance of the best
retrieved document, a signal already available from the retrieval pipeline.

\subsubsection{Routing Logic}

The hybrid engine calls the same two-stage retrieval as the RAG system, then
selects a generation strategy based on the best distance (excluding
companion documents):

\begin{description}
    \item[Case~A: Strong retrieval] ($d \leq 1.2$).
    Uses the RAG strategy: the model relies on retrieved context, mimics the
    correction style, and cites sources. Same anti-hallucination rules as
    standalone RAG.

    \item[Case~B: Partial retrieval] ($1.2 < d \leq 1.6$).
    Includes retrieved documents \textit{and} supplements the prompt with the
    curriculum inventory. The model is told to use documents when pertinent,
    complement with curriculum knowledge where gaps exist, and signal which
    parts come from which source.

    \item[Case~C: No useful retrieval] ($d > 1.6$).
    Discards all retrieved documents and falls back to the prompt-only
    strategy.
\end{description}

This is a discrete approximation of the retrieval quality assessment in
CRAG~\cite{yan2024practical_rag}, using two fixed thresholds instead of a
learned classifier.

\subsubsection{Thresholds and Confidence}

The thresholds (1.2 and 1.6) were calibrated empirically during development
by examining L2 distance distributions across test queries. They are not
claimed to generalize beyond this corpus and embedding model.

Confidence depends on the routing case: Case~A uses the RAG formula
(distance + context volume), Case~B is capped at ``medium'' (since
partially grounded answers should not claim high confidence), and Case~C
falls back to the keyword heuristic.

\subsubsection{Relationship to Adaptive Retrieval}

Compared to CRAG (which can trigger web searches) and Self-RAG (which
trains the model to emit retrieval tokens), our routing is deliberately
minimal---a simple distance threshold check. This simplicity is appropriate
for a fixed, domain-specific corpus. The contribution is not the routing
mechanism itself but the observation that even simple quality-aware routing
can avoid the failure modes of both pure-RAG and pure-prompting.
